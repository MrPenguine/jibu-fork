## 1. INTRODUCTION TO SERVICES

Services in the Jibu Console backend are responsible for implementing the business logic of the application. They act as an intermediary between controllers and data access layers, encapsulating complex operations and ensuring data consistency. Services are implemented using NestJS's dependency injection system, making them highly testable and maintainable.

The primary responsibilities of services include:
- Implementing business logic and domain rules
- Coordinating between multiple repositories or other services
- Handling domain-specific validation
- Managing transactions and ensuring data consistency
- Implementing caching strategies where appropriate
- Integrating with external systems and APIs

## 2. SERVICE IMPLEMENTATION PATTERN

Services in Jibu Console follow a consistent implementation pattern:

1. Class Declaration with @Injectable Decorator:
```typescript
@Injectable()
export class ResourceService {
  constructor(
    @InjectRepository(ResourceEntity)
    private readonly resourceRepository: Repository<ResourceEntity>,
    private readonly otherService: OtherService,
    private readonly configService: ConfigService,
  ) {}
}
```

2. CRUD Operations:
```typescript
async create(createResourceDto: CreateResourceDto): Promise<Resource> {
  const resource = this.resourceRepository.create(createResourceDto);
  return this.resourceRepository.save(resource);
}

async findAll(options: PaginationQueryDto): Promise<PaginatedResponse<Resource>> {
  const [items, totalItems] = await this.resourceRepository.findAndCount({
    skip: (options.page - 1) * options.limit,
    take: options.limit,
    where: options.filter,
    order: options.sort,
  });

  return {
    items,
    meta: {
      totalItems,
      itemCount: items.length,
      itemsPerPage: options.limit,
      totalPages: Math.ceil(totalItems / options.limit),
      currentPage: options.page,
    },
  };
}

async findOne(id: string): Promise<Resource> {
  const resource = await this.resourceRepository.findOne({ where: { id } });
  if (!resource) {
    throw new ResourceNotFoundException('Resource', id);
  }
  return resource;
}

async update(id: string, updateResourceDto: UpdateResourceDto): Promise<Resource> {
  const resource = await this.findOne(id);
  const updatedResource = this.resourceRepository.merge(resource, updateResourceDto);
  return this.resourceRepository.save(updatedResource);
}

async remove(id: string): Promise<void> {
  const resource = await this.findOne(id);
  await this.resourceRepository.remove(resource);
}
```

3. Business Logic Methods:
```typescript
async processResource(id: string, data: ProcessDataDto): Promise<ProcessResult> {
  const resource = await this.findOne(id);
  
  // Business logic implementation
  const result = await this.performComplexOperation(resource, data);
  
  // Update resource state
  resource.status = 'processed';
  await this.resourceRepository.save(resource);
  
  return result;
}

private async performComplexOperation(resource: Resource, data: ProcessDataDto): Promise<ProcessResult> {
  // Complex business logic implementation
}
```

4. Integration with Other Services:
```typescript
async integrateWithExternalSystem(id: string): Promise<IntegrationResult> {
  const resource = await this.findOne(id);
  
  // Call external service
  const externalData = await this.otherService.fetchExternalData(resource.externalId);
  
  // Process and integrate data
  const integrationResult = this.processExternalData(resource, externalData);
  
  // Update resource with integration results
  resource.integrationStatus = 'completed';
  resource.lastIntegrationDate = new Date();
  await this.resourceRepository.save(resource);
  
  return integrationResult;
}
```

5. Caching Strategies:
```typescript
@Cacheable('resource', { ttl: 60 })
async findOne(id: string): Promise<Resource> {
  const resource = await this.resourceRepository.findOne({ where: { id } });
  if (!resource) {
    throw new ResourceNotFoundException('Resource', id);
  }
  return resource;
}

@CacheEvict('resource')
async update(id: string, updateResourceDto: UpdateResourceDto): Promise<Resource> {
  const resource = await this.findOne(id);
  const updatedResource = this.resourceRepository.merge(resource, updateResourceDto);
  return this.resourceRepository.save(updatedResource);
}
```

## 3. CORE SERVICES

### 3.1. API Key Service

The API Key Service manages the creation, retrieval, updating, and deletion of API keys. API keys are used for authenticating API requests from external systems.

File: apps/backend/src/modules/v1/api-key/api-key.service.ts

Key Methods:
- createApiKey(createApiKeyDto): Creates a new API key
- findAll(options): Retrieves all API keys with pagination and filtering
- findOne(id): Retrieves a single API key by ID
- update(id, updateApiKeyDto): Updates an API key's metadata
- remove(id): Deletes an API key
- validateApiKey(key): Validates an API key and returns the associated user and organization
- rotateApiKey(id): Generates a new API key value while preserving metadata

Implementation Details:
- Uses TypeORM for data access
- Implements secure generation and storage of API keys
- Supports filtering by organization ID
- Implements caching for frequently accessed API keys
- Handles API key rotation with proper invalidation of old keys
- Integrates with the user and organization services for validation

### 3.3. Chats Service

The Chats Service manages chat sessions and message history. It provides functionality for creating chat sessions, sending and receiving messages, and retrieving chat history.

File: apps/backend/src/modules/v1/chats/chats.service.ts

Key Methods:
- createChat(createChatDto): Creates a new chat session
- findAll(options): Retrieves all chats with pagination and filtering
- findOne(id): Retrieves a single chat by ID
- update(id, updateChatDto): Updates a chat session
- remove(id): Deletes a chat session
- sendMessage(id, messageDto): Sends a message in a chat session
- getChatHistory(id): Retrieves the message history for a chat session

Implementation Details:
- Uses TypeORM for data access
- Implements a Redis-based chat history system with dual key patterns:
  1. `chat:history:{chatId}` - Direct access by chat ID
  2. `chat:{assistantId}:{sessionId}` - For backward compatibility
- Includes a local caching mechanism with 5-second TTL for performance
- Integrates with the Assistant Service for AI-powered responses
- Supports file attachments and rich message formats
- Implements real-time message streaming using Server-Sent Events (SSE)

### 4.2. Redis Integration

The Redis integration provides services for caching, pub/sub messaging, and data storage using Redis. This includes the RedisService and various Redis-based implementations.

File: apps/backend/src/core/redis/redis.service.ts

Key Methods:
- get(key): Gets a value from Redis
- set(key, value, expirySeconds): Sets a value in Redis with optional expiration
- del(key): Deletes a key from Redis
- exists(key): Checks if a key exists in Redis
- hset(key, field, value): Sets a hash field in Redis
- hget(key, field): Gets a hash field from Redis
- hgetall(key): Gets all fields and values from a hash in Redis
- lrange(key, start, stop): Gets elements from a list in Redis
- rpush(key, value): Adds a value to the end of a list in Redis

Key Features:
- Caching with configurable TTL
- Pub/sub messaging for real-time communication
- Data storage for transient data
- Rate limiting and throttling
- Session management

Implementation Details:
- Uses the ioredis client for Node.js
- Implements connection pooling for efficient resource usage
- Provides serialization and deserialization of complex objects
- Handles connection errors and retries
- Implements cache eviction strategies
- Comprehensive error handling and logging

The Redis integration is used throughout the application for various purposes, including:
- Chat history storage and retrieval through dual key patterns
- Caching of frequently accessed data with configurable TTL
- Real-time message streaming for chat applications
- Rate limiting for API endpoints to prevent abuse
- Session management for authentication and user state
- Distributed locking for concurrent operations

The Redis-based chat history system is a key component of the Jibu Console, providing efficient storage and retrieval of chat messages for assistants. It implements a dual key pattern for flexible access patterns and includes a local caching mechanism for improved performance.
